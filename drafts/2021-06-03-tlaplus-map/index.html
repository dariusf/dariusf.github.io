A map of the TLA+ ecosystem

3 Jun, 2021

tla cheat sheet

possible blog post it’s a highly coherent, well thught and useful project. unfortunately this coherence is spread across many books and papers written over decades. even sriram says read it several different ordeers until you get it, no kidding. even the pluscal book doesn’t do it for me because it focuses too much on surface syntax and use cases, whereas this overiew is organised by concept

a map. by topic. for navigating. pointers to the best resource

fairness. in specifying systems probably

deadlock: will be checked for when spec has no more transitions. to disable, add an no-op unchanged-everything action

termination: caused by the pluscal translation. generates an extra label with done, and an extra def for termination which checks it eventually, then adds termination as a temporal logic PROPERTY in the cfg file

PROPERTY: for temporal properties. tends to be slow

fairness: this is a constraint that removes some states. so it’s a conjunct of the spec, init /\ next /\ fairness

constraint, action constraint: ways to remove states. also used to make a spec finite domain for model checking

assume: not sure. i think it’s to assert things about model values? provided in cfg file for model checking

model values: provided in cfg file. will not be equal to others. to turn this into a noisy failure there are typed model values, which are a nasty hack using names. i found this out the hard way when generating tla+ code

tla2tools: add some notes about the java interface

development environment: vscode. debugger. trace debugging

extra logical featueres in tlc and ways to do debugging

tla2json ecosysem animation?

3 things that can be checked. invariants, deadlock, temporal properties

as with any less-than-mainstream language, it’s not easy to simply google your problems when using TLA+ and PlusCal. the frustrating part is finding solutions in the many documents that have been released over the years.

there is no fault of the community: it’s genuinely difficult to document every quirk of a large project like TLA+, particularly one that has spanned several years. but with sufficient dedication this can be overcome.

this post intends to serve as a map to the ecosystem. minimally, it’ll serve a small collection of bookmarks so you can quickly get what you need to go.

of course, such a list is only useful if you know where to go. for that i have put an index at the bottom, with the best reference i know of.
